# Authentication Flow Requirements Implementation

## Overview
This document outlines the security enhancements made to the Student Housing Booking System to enforce strict authentication and session management rules.

## Requirements Enforced

### 1. Login Flow
? **Requirement**: User must perform a successful Login first to obtain an access token and a refresh token.

**Implementation**:
- `LoginAsync` validates credentials and creates a new user session
- Both access token and refresh token are generated and returned
- Refresh token is **persisted to the database** with:
  - Token value
  - User ID (for validation)
  - Expiry date (7 days)
  - Created timestamp
  - Revoked flag (initially false)

### 2. Token Refresh Flow
? **Requirement**: After access token expires, the system allows generating a new access token using the refresh token.

**Implementation**:
- `RefreshTokenAsync` now **validates refresh token against the database** before issuing new tokens:
  - Checks that refresh token is not revoked
  - Checks that refresh token is not expired
  - Checks that refresh token belongs to the authenticated user (matches UserId from access token)
  - Checks that refresh token is not blacklisted in memory
  - Validates access token signature and user claims
- New refresh token is generated and persisted to database
- Both tokens are returned if validation succeeds

### 3. Logout Flow
? **Requirement**: User must perform Logout by sending the valid token to properly terminate the session.

**Implementation**:
- `LogoutAsync` requires **both access token and refresh token** (enforced by validator)
- Validates that:
  - Access token is valid and can be parsed
  - Refresh token exists in database
  - Refresh token belongs to the same user (UserId matches)
  - Both tokens have not been previously revoked
- Upon successful validation:
  - Marks the specific refresh token as revoked in database with timestamp
  - Revokes ALL other refresh tokens for the user (prevents token reuse attacks)
  - Blacklists both tokens in memory
  - Returns true only if all steps succeed

### 4. Forbidden Actions
? **All forbidden actions are now prevented**:

#### a) Logout without valid token
- Logout endpoint requires both tokens (FluentValidation)
- If either token is missing, request fails validation
- If tokens are invalid, LogoutAsync returns false ? BadRequest response

#### b) Generate refresh token before Login
- Refresh tokens are only generated by:
  - `LoginAsync` - requires successful authentication
  - `RegisterStudentAsync` - requires successful registration
  - `RefreshTokenAsync` - requires valid existing tokens

#### c) Generate refresh token after Logout
- Once logout succeeds, both tokens are revoked in database
- Attempting to use revoked refresh token in `RefreshTokenAsync` returns error: "Invalid or expired refresh token"
- Attempting to logout again fails validation (tokens already revoked)

### 5. Post-Logout Token Invalidation
? **All tokens are properly invalidated after logout**:

- **Database Persistence**: 
  - Specific refresh token marked as `IsRevoked = true` with `RevokedAt` timestamp
  - All other user refresh tokens revoked to prevent session hijacking

- **In-Memory Blacklist**:
  - Both access and refresh tokens added to blacklist
  - Any attempt to use blacklisted tokens is rejected immediately

- **Validation Flow**:
  - `RefreshTokenAsync` checks both database and in-memory blacklist
  - Revoked tokens cannot be used to generate new tokens

## Technical Implementation Details

### Database Changes
- Added `DbSet<RefreshToken>` to `StudentHousingDBContext`
- `RefreshToken` entity stores:
  - `Id` - Primary key
  - `Token` - The token value
  - `UserId` - User ID (foreign key)
  - `IsRevoked` - Revocation status
  - `ExpiryDate` - Token expiration date
  - `CreatedAt` - Creation timestamp
  - `RevokedAt` - Revocation timestamp (nullable)

### Service Layer
- **AuthService**:
  - Injected `IRefreshTokenRepository` dependency
  - All authentication methods now persist/validate refresh tokens
  - Comprehensive error handling with specific error messages

- **TokenBlacklistService**:
  - In-memory token blacklist for immediate rejection
  - Supplements database validation for performance

### Repository Layer
- **RefreshTokenRepository**:
  - `GetValidTokenAsync()` - Fetches non-revoked, non-expired tokens
  - `RevokeAllUserTokensAsync()` - Invalidates all user tokens on logout
  - Used for database-level token validation

### Validation Layer
- **LogoutValidator**:
  - Requires both tokens to be non-empty
  - Prevents logout without valid tokens

- **RefreshTokenValidator**:
  - Requires both tokens to be non-empty

## Security Improvements

1. **Refresh Token Storage**: Tokens no longer exist only in memory; they're persisted to database with metadata
2. **Token Revocation**: Explicit revocation tracking with timestamps for audit trails
3. **Session Hijacking Prevention**: All user tokens revoked on logout prevents token reuse
4. **Comprehensive Validation**: Multi-layer validation (format, blacklist, database, ownership)
5. **Clear Error Messages**: Specific error responses help prevent information leakage while aiding debugging
6. **User Identity Verification**: Refresh token ownership verified by matching UserId from access token

## API Contract

### Login
```
POST /api/v1/account/login
Request: { email, password }
Response: { token: { accessToken, refreshToken }, user }
```

### Register
```
POST /api/v1/account/register/student (or /landlord)
Request: { email, password, ... }
Response: { token: { accessToken, refreshToken }, user }
```

### Refresh Token
```
POST /api/v1/account/refresh-token
Request: { token (access token), refreshToken }
Response: { token: { accessToken (new), refreshToken (new) } }
Requirements:
- Access token must be valid (parseable)
- Refresh token must exist in database, not revoked, not expired
- Refresh token must belong to the authenticated user
```

### Logout
```
POST /api/v1/account/logout
Request: { token (access token), refreshToken }
Requirements:
- Both tokens MUST be provided
- Access token must be valid
- Refresh token must exist in database and not already revoked
- Tokens must belong to the same user
Behavior:
- Revokes the refresh token in database
- Revokes all other refresh tokens for the user
- Blacklists both tokens in memory
```

## Compliance Checklist

- [x] Login generates and stores refresh token
- [x] Refresh token can only be used to generate new tokens (not generated directly)
- [x] Logout requires valid tokens
- [x] Logout invalidates all user tokens
- [x] Refresh tokens expire after 7 days
- [x] Revoked tokens cannot be reused
- [x] Multiple token refresh attempts are tracked
- [x] Logout cannot proceed without valid token
- [x] All validation failures return clear error messages
- [x] Database changes tracked (with timestamps)
